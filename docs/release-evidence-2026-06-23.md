Docker CLI, Compose, and Docker Desktop daemon were reachable during final verification: Compose config passed, the local image built, and container runtime smokes passed for the CLI, doctor, relay, gateway, native host, and verifier entrypoints.

`npm run release:audit` is the repeatable local package/demo evidence command. It runs the local package gates and smokes below, prints an `enigma.release_audit.v1` JSON summary, supports `-- --out <file>` for handoff/goal-audit ingestion, and fails nonzero on any failed required local gate without requiring Docker, cloud credentials, npm publish credentials, or a live website; the separate Docker compose/build/runtime smoke recorded below passed once Docker Desktop was reachable.

`npm run provenance:local -- --out ./.enigma/release-provenance.json` is the local provenance/SBOM checksum evidence command. It records package-surface file inventory and SHA-256 values for reviewer comparison. It is not signed provenance, registry attestation, source-control commit evidence, SLSA/compliance evidence, Docker image digest evidence, or hosted/cloud deployment evidence.

`npm run review:packet -- --out ./.enigma-review-packet --public-site <path-to-_public_site>` is the local hand-review bundle command. It packages release-audit, provenance, package-review, selected backend source/test files, production validators, public API docs, whitepaper evidence, and optional generated public-site evidence behind a manifest so reviewers can inspect files and SHA-256 values without reconstructing commands. Docker image/runtime proof is the separate compose/build/runtime evidence recorded below. It is not npm publication, live Cloudflare deployment, signed provenance, hosted/BYOC readiness, legal approval, compliance status, or proof that external providers deleted or forgot data.

`enigma native-host install-plan --browser <chrome|edge|firefox> --manifest <absolute path> [--os <windows|macos|linux>] [--home <absolute path>]` is the local native-host installation planner. It prints exact target manifest paths, manual steps, Windows registry command previews when applicable, and `writes_performed: false`. It does not copy manifests, write browser profile files, write Windows registry keys, or complete native-host registration; user/operator execution remains required.

`python scripts/preflight_public_site.py --site _public_site` is the credential-free public-site artifact preflight before Cloudflare handoff. It reads the local generated artifact and reports checked items, counts, warnings, and blockers. It does not require Cloudflare credentials, deploy files, configure DNS/TLS, change cache state, or prove live availability.

`npm run infrastructure:readiness -- --manifest <path>` is the local infrastructure readiness evidence command. Without `--live`, it emits `enigma.infrastructure_readiness.v1` contract evidence for an `enigma.infrastructure_readiness_manifest.v1` manifest with `schema`, `ok`, `generated_at`, `mode`, `credentials_required:false`, `credentials_used`, readiness booleans, checks, blockers, and claim boundary. With `--live`, it is public endpoint evidence only. `--cloudflare-live off|auto|required` is optional Cloudflare token/account/static-site observation only; it is not hosted relay/gateway, KMS, storage, SIEM, backup, operator, or token-launch readiness.

## CI gate interpretation

CI is package-gate evidence for the same local commands listed above. It intentionally does not require Docker daemon access, cloud/deploy credentials, npm publish credentials, domains, KMS/secrets, SIEM routes, support/SLA approvals, legal/compliance approvals, or a completed operator acceptance packet. A green CI run does not replace manual website review, deployment readiness review, legal/compliance review, or hosted/BYOC operator acceptance.

## Local release audit interpretation

`npm run release:audit` reproduces the local package/demo evidence path only. It may include optional local preflights such as native-host install-plan shape checks and public-site artifact checks when their local inputs exist, but those preflights are no-write/credential-free planning evidence. The audit intentionally remains independent of Docker daemon runtime, which is covered by the separate compose/build/runtime smoke rows below.

The audit also excludes hosted relay/gateway deployment, live Cloudflare/domain verification, npm publication, customer BYOC acceptance, legal/compliance approval, live website review, and operator acceptance packet completion because those require external infrastructure, credentials, or operator sign-off.

`npm run infrastructure:readiness -- --manifest <path>` is narrower than release audit: it records local infrastructure-contract readiness and optional public endpoint observation for the manifest under review. It does not deploy, mutate Cloudflare/DNS, provision KMS/storage/SIEM/backup, create tokens, or accept hosted/BYOC operations.

## Local provenance/SBOM interpretation

`npm run provenance:local -- --out ./.enigma/release-provenance.json` can be used by a reviewer to create a local checksum/SBOM evidence file for the exact package-surface files inventoried in the reviewed checkout or artifact directory. Reviewers compare recorded `(path, sha256)` entries with freshly computed SHA-256 values or with another provenance file to identify local file drift.

This evidence is complementary to `npm pack --dry-run`: the dry-run previews the npm tarball contents, while local provenance records checksums for the package-surface files the provenance command inventories. If the public site artifact is in scope, build the artifact first and then rerun local provenance so the optional `public_site_manifest` checksum reflects the local generated manifest when present. That still proves only local manifest bytes; it does not prove Cloudflare deployment, DNS/TLS, cache state, or live availability.

Do not use local provenance/SBOM output as evidence of signed attestation, registry provenance, git commit identity, clean source-control state, SLSA level, compliance status, Docker image digest/runtime, hosted/BYOC deployment, or third-party provider deletion/model forgetting.

## Reviewer packet interpretation

`npm run review:packet -- --out ./.enigma-review-packet --public-site <path-to-_public_site>` creates a local review directory for human inspection. Reviewers should open `.enigma-review-packet/REVIEW_PACKET_MANIFEST.json` first, inspect the included paths and `sha256:<hex>` values, and then review `evidence/release-audit.json`, `evidence/local-provenance.json`, `package/npm-pack-dry-run.json`, copied `docs/`, selected `source/` files, and optional `site/` contents listed by that manifest.

The packet is an evidence index and convenience bundle. It does not replace `npm run release:audit`, `npm run provenance:local`, public-site build review, Docker image/runtime smoke, CI policy review, npm publication, Cloudflare deployment verification, legal/compliance approval, or hosted/BYOC operator acceptance.

Any other missing external evidence remains a blocker and must be recorded as such.

## Backend changes covered by tests

- Strict public receipt verification rejects unsigned extension fields, plaintext-shaped receipt keys, and signature extension fields.
- Vault records no longer keep enumerable plaintext after remember/update/recall/delete/export.
- Receipt `receipt_log_root` semantics are now prefix-before-current, avoiding circular self-inclusion while still verifying real vault receipt streams.
- Public/export/context surfaces sanitize normalized plaintext-like keys and source references, including camelCase variants such as `rawMemory`.
- Context-pack verification returns redacted public output while private context remains available for local injection.
- Context-pack public verification now requires explicit public-key/trust material before returning `valid:true`; no-key checks are shape/status only.
- MCP now validates protocol versions, ids, params, tool args, notifications, batches, and error codes.
- Connector writes are semantic, idempotent, backed up only on real changes, and written atomically.
- Vault metadata is allowlisted/committed before storage/export so raw memory cannot leak through keys such as `summary`, `description`, `note`, unknown primitives, nested metadata, or camelCase variants.
- Importer/capsule verification checks manifest/payload/verifier metadata/trust before vault writes; failed imports write nothing; capsule custody roots require embedded matching receipt objects, not hash-only vault-write claims.
- CLI/MCP bundle verification is bound to committed exported vault roots and sequence, so truncated but internally valid receipt prefixes fail.
- Relay/mesh reject plaintext-like payloads and store hash-only public relay records.
